# 임베딩 노트 (Phase 3 / 041)

152,611 child를 Qwen3-Embedding-0.6B로 로컬(Apple Silicon, MPS) 임베딩하며 겪은 속도 이슈와 조정.

## 문제 — 초기 설정이 느림
초기: fp32 / batch 64 / `max_seq_length=2048`. 13.5분 동안 첫 shard(10k)도 못 끝냄
→ 실측 환산 **~12 chunks/sec → 전체 ~3시간**. CPU 6%(=MPS-bound지만 처리량 낮음).

## 조정
| 항목 | before | after | 이유 |
|---|---|---|---|
| dtype | fp32 | **fp16** (`torch_dtype=float16`) | MPS 연산량 절반, 임베딩 정밀도 손실 무시 가능 |
| batch | 64 | **128** | MPS GPU 활용률↑ |
| max_seq_length | 2048 | **512** | 청크 p95=382라 대부분 안 잘림. 512+ 청크 134개(0.09%)만 살짝 절단(긴 나열문 — 저가치) |

## 결과 — 조정은 미미, MPS가 병목
2000 청크 벤치: **fp16 후 16 chunks/sec** (before ~12/sec). **전체 152,611 → ~2.7시간(163분).**
즉 fp16/batch/seq 조정은 ~33%만 개선했고, 근본 병목은 **PyTorch MPS가 이 모델에 느린 것**
(chunk당 ~62ms, GPU 가속이 약함). 설정 문제 아님.

## 대안 (속도 우선이면)
- **Ollama(Metal GGUF)** — 보통 PyTorch MPS보다 빠름. 단 런타임 전환 + 양자화로 임베딩이 미세하게
  달라질 수 있어 0.6B 전체/4B·8B 비교를 같은 런타임으로 통일하는 게 깔끔.
- **그냥 ~2.7h 백그라운드 1회** — 일회성이고 무인 실행. 런타임 일관성 유지. (← 채택)

## 트레이드오프
- fp16: 임베딩 코사인 검색엔 영향 거의 없음(표준 관행).
- max_seq_length 512: 0.09% 청크 꼬리 절단. 해당 청크는 문장경계 없는 긴 나열문이라 검색가치 낮음.
- 이 값들은 0.6B 전체 임베딩용. 4B/8B 평가셋 비교 시 동일 원칙 적용.
