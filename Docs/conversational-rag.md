# 멀티턴 대화 (conversational RAG)

챗봇이 이전 대화를 기억해서 후속 질문을 이해하게 만든 방식 정리.

## 문제

RAG는 기본적으로 한 질문씩 독립으로 처리한다. 그래서 "카뮈의 부조리가 뭐야?"를 묻고 나서
"그럼 이걸 공리주의로 보면?"이라고 이어 물으면, 시스템 입장에선 "이걸"이 뭔지 모른다.
그 문장만 그대로 벡터 검색에 넣으면 엉뚱한 항목이 잡히고, 생성 모델도 맥락 없이 답한다.

## 해결 — 맥락을 두 군데서 쓴다

대화 기록은 `st.session_state.messages`에 들고 있고(브라우저 세션 한정, "새 세션"으로 초기화),
질문이 들어올 때마다 그 직전까지의 기록을 두 곳에 넘긴다.

### 1. 검색 전 — 질문 맥락화 (`qa.condense_query`)

후속 질문을 **직전 대화를 반영한 독립적인 검색 쿼리로 다시 쓴다.**

```
대화: User: 카뮈의 부조리?  / Assistant: 부조리는 의미를 찾는 인간과 침묵하는 세계의 충돌…
후속: 그럼 이걸 공리주의로 보면?
  ↓ condense_query (gemma3:4b)
검색 쿼리: "How does Camus's concept of the absurd relate to utilitarianism?"
```

- 최근 6턴 + 후속 질문을 작은 모델에 주고 "맥락을 채운 standalone 영어 검색 쿼리 한 줄"로 받는다.
- **가벼운 모델(gemma3:4b)** 로 빠르게. 최종 답변용 큰 모델은 안 건드린다(지연 최소화).
- 첫 질문(대화 기록 없음)이면 그냥 원문 그대로 통과.
- 코퍼스가 영어라 검색 쿼리는 영어로 뽑는다. 한국어/중국어로 물어도 검색은 영어로 매칭되고,
  UI엔 재작성된 쿼리를 `🔎` 캡션으로 보여줘 무엇이 검색됐는지 드러낸다.

### 2. 생성 시 — 대화 맥락 주입 (`qa.build_prompt(history)`)

검색해 온 SEP 컨텍스트와 함께 **직전 대화를 프롬프트에 같이 넣는다.** 시스템 프롬프트가
"this/it 같은 지시어는 대화로 해소하고, 질문과 같은 언어로 답하라"고 지시한다. 그래서
답변은 질문한 언어(한국어/중국어/영어)로 돌아오고, 출처는 영어 SEP 딥링크로 붙는다.

## 전체 흐름 (`08_chatbot/app.py`)

```
사용자 입력
  → history = 직전 메시지들 캡처
  → search_q = condense_query(history, 입력)        # 맥락화(작은 모델)
  → retrieve(search_q)                              # dense→rerank→small-to-big→(graph)
  → ollama(model, build_prompt(입력, parents, history), think=…)   # 생성(큰 모델)
  → 답변 + thinking(옵션) + 출처 expander
  → messages에 누적
```

## 설계 메모

- 이 구조는 LangChain의 *history-aware retriever*(질문 재작성 체인)와 같은 패턴을 직접 구현한 것.
- 프롬프트에 넣는 대화는 **최근 6턴**으로 제한. 길어지면 요약 압축을 도입할 여지가 있다(현재는 미적용).
- 검색용 재작성과 답변 생성을 **다른 모델**로 나눈 게 핵심 — 재작성은 싸고 빠르게, 답변만 품질 모델로.
- 메모리는 세션 메모리(영속 X). 멀티유저/영속 대화가 필요하면 별도 store가 필요하다(범위 밖).
